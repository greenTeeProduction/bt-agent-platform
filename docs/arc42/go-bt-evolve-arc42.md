3|## 1.1 Requirements Overview
4|
5|go-bt-evolve is a Go behavior tree agent platform that provides:
6|
7|- **BT Execution Engine** вҖ” Builds and executes behavior trees with Sequence, Selector, Action, Condition, ChainAction, UtilitySelector, PlannerNode, and event-driven node types.
8|- **41 Trees across 7 Categories** вҖ” domain (code_review, devops_ci, agent_monitor, refactoring, security_audit, data_pipeline, meeting_notes, crash_investigator, game_ai, trading_signal), finance (10 trees), research (deep_research, quick_research), startup (8 roles), thinktank (synthesis, peer_review, report), evolution (stockfish, hermes_evolve), core.
9|- **3 MCP Servers** вҖ” bt-agent (36 tools), bt-evaluator (5 tools), bt-langagent (2 tools), all via JSON-RPC 2.0 over stdio.
10|- **Dashboard** вҖ” HTTP server on :9800 with 8 tabs (Overview, ThinkTank, Company, Tasks, Tree View, Evolution, Agents, MindMap).
11|- **Evolution Engine** вҖ” Stockfish-adapted mutation ordering, Pareto multi-objective front, MAP-Elites quality diversity, Island Model with migration, Q-Learning epsilon-greedy.
12|- **Agent Platform** вҖ” YAML-defined agents with registry, scheduler, circuit breakers, dead letter queue, A2A (Agent-to-Agent) protocol, memory store, and webhook publishing.
13|- **Knowledge Graph** вҖ” Semantic index of all trees with embeddings, capabilities, and cross-tree relationships for discovery and auto-creation.
14|- **Factory** вҖ” Tree breeding via crossover from parent templates, archetype-based generation.

1|# arc42 Section 8 вҖ” Crosscutting Concepts
2|
3|## 8.1 Behavior Tree Execution Model
4|
5|**What:** All agent logic is expressed as behavior trees вҖ” hierarchical structures of Sequence, Selector, Action, Condition, ChainAction, and decorator nodes. ADR-001.
6|
7|**Why:** BT nodes are composable, testable, and evolvable. The tick-based execution model supports multi-step planning with interleaved LLM calls. StrategyRouter + OutcomeSelector provide structured fallback.
8|
9|**Where:** `internal/engine/tree.go` (BuildTree, RunTask, Blackboard), `internal/engine/registry.go` (action/condition registration), `internal/domains/` (tree definitions).
10|
11|**Effect:** Every task flows through PreGateвҶ’StrategyRouterвҶ’OutcomeSelector. New capabilities are added by registering actions вҖ” no control flow changes needed.
12|
13|## 8.2 ChainAction Nodes
14|
15|**What:** LLM calls wrapped as behavior tree leaf nodes. 10 chain types (`llm_call`, `agent`, `rag_query`, `tool_call`, `structured_output`, `refine`, `map_reduce`, `conversation`, `retrieval_qa`, `tool_action`). ADR-006.
16|
17|**Why:** Integrating LLM calls as BT nodes enables PreGate gating (don't call LLM if preconditions fail), retry wrapping, and StrategyRouter selection (try primary prompt, fall back to alternate prompt).
18|
19|**Where:** `internal/engine/chains.go`. Config read from node `Name` (format: `chain_type:prompt_text`) and `Metadata` (max_tokens, temperature).
20|
21|**Template variables:** `{{.Task}}`, `{{.Plan}}`, `{{.Result}}`, `{{.CachedResult}}`, `{{.ChainState.*}}`, `{{.ChainMemory}}`, `{{.ChainTools}}`, `{{.KgResults}}`.
22|
23|**Effect:** LLM integration is a first-class BT concept, not a side-effect. Chains can be retried, gated, and selected just like any other action.
24|
25|## 8.3 MCP Protocol Layer
26|
27|**What:** All tools are exposed via JSON-RPC 2.0 over stdio. 3 MCP servers: bt-agent (36 tools), bt-evaluator (5 tools), bt-langagent (2 tools). ADR-002.
28|
29|**Why:** MCP provides a standardized interface between Hermes Agent and the Go BT platform. No custom protocols, no REST overhead. Stdio transport keeps it simple and gateway-managed.
30|
31|**Where:** `internal/mcp/` (server implementation), `cmd/bt-agent/tools.go` (tool registration), `cmd/bt-agent/main.go` (server setup).
32|
33|**Effect:** Hermes Agent sees 43 MCP tools. Adding a tool is a single `server.RegisterTool()` call. Gateway handles lifecycle (spawn, restart, health check).
34|
35|## 8.4 File-Based Persistence
36|
37|**What:** All state stored as JSON/YAML files with atomic writes (write .tmp вҶ’ rename). No SQL database. ADR-003.
38|
39|**Why:** Git-friendly (diffs are readable), no database dependency, single-file atomicity prevents corruption. Simpler than SQL for a single-machine platform.
40|
41|**Where:** `~/.go-bt-evolve/` directory tree. Agent YAMLs, scheduler JSON, history JSON, reflection records, DLQ JSON, tree store JSON.
42|
43|**Effect:** State survives restarts. Git can version agent definitions. Manual inspection and repair is possible with any text editor.
44|
45|## 8.5 Evolution Pipeline
46|
47|**What:** Common pattern for tree improvement: evaluate вҶ’ order mutations вҶ’ apply top mutation вҶ’ re-evaluate вҶ’ compare fitness вҶ’ accept (commit) or rollback.
48|
49|**Why:** Multiple algorithms (Stockfish, Pareto, MAP-Elites, Island, Q-Learning, Expert) share this pattern. A unified pipeline reduces duplication and ensures consistent safety checks.
50|
51|**Where:** `internal/evolution/` вҖ” each algorithm file, `internal/gardener/` (evolution_v2.go for cycle orchestration).
52|
53|**Effect:** Evolution is auditable (git commits), reversible (rollback), and measurable (fitness delta tracking).
54|
55|## 8.6 Error Resiliency
56|
57|**What:** SafeGo (panic recovery) + CircuitBreaker (3-state) + RetryWithBackoff (full jitter, 3 classes) + DeadLetterQueue (persistent JSON). ADR-007.
58|
59|**Why:** LLM calls can fail (Ollama OOM, DeepSeek rate limits, network timeouts). Goroutines must not crash the process. Failed work must not be silently lost.
60|
61|**Where:** `internal/reliability/` вҖ” SafeGo, CircuitBreaker, RetryPolicy, DeadLetterQueue. Applied in scheduler runner (`main.go:276`), ChainAction execution, and all goroutine spawns.
62|
63|**Effect:** The platform degrades gracefully. A single LLM failure doesn't cascade. Failed tasks are preserved for inspection.
64|
65|## 8.7 Quality Gates
66|
67|**What:** Output validation before declaring success. Minimum length, error pattern detection, structure scoring (markdown, bullets, code blocks). QualityScore 0.0-1.0.
68|
69|**Why:** LLMs sometimes produce truncated/garbage output (e.g., max_tokens=10 producing a few words). Without validation, agents report "success" with useless output.
70|
71|**Where:** `internal/engine/tree.go:validateOutputQuality()`. Applied after every RunTask() and in ReflectOnOutcome action.
72|
73|**Effect:** Structured zero-LLM output (alert_router, agent_monitor) scores correctly. Garbage output is flagged. Quality scores feed into fitness evaluation.
74|
75|## 8.8 Tool Protocol
76|
77|**What:** ChainAction nodes use tool stubs (Name/Description/Call) populated at PreGate. Tools inject file I/O, shell execution, and codebase inspection capabilities into LLM chains.
78|
79|**Why:** LLMs need access to real tools (read files, run commands, query the graph) during chain execution. Tool stubs provide a uniform interface.
80|
81|**Where:** `internal/engine/tools_real.go` (real implementations), `internal/engine/tree.go:toolStub` (lightweight wrapper). Setup actions: `SetupDefaultTools`, `SetupDevTools`, `SetupResearchTools`.
82|
83|**Effect:** ChainAction prompts can reference `{{.ChainTools}}` and the LLM can reason about available capabilities.

1|# arc42 Section 9 вҖ” Architecture Decisions
2|
3|## ADR-001: Behavior Trees as Core Execution Model
4|
5|**Context:** We needed a deterministic, composable execution model for AI agent workflows. Simple linear scripts couldn't handle branching, retry, or fallback. Full GOAP was over-engineered for most tasks.
6|
7|**Decision:** Use behavior trees with Sequence, Selector, Action, Condition, ChainAction, and decorator nodes. All agent logic is a tree. The Blackboard pattern carries shared state through ticks.
8|
9|**Status:** Accepted (2026-05-26)
10|
11|**Consequences:**
12|- вң… Composable: PreGateвҶ’StrategyRouterвҶ’OutcomeSelector is the universal pattern
13|- вң… Evolvable: Trees are data вҖ” mutation, crossover, and versioning work naturally
14|- вң… Testable: Each node type has clear contracts
15|- вҡ пёҸ Learning curve: Developers must understand BT semantics (ticks, Running state, Selector fallthrough)
16|- вҡ пёҸ 1000-tick safety limit can abort long-running agent loops
17|
18|## ADR-002: MCP as External Interface
19|
20|**Context:** Hermes Agent is a Python process. The BT platform is Go. We needed a protocol for them to communicate.
21|
22|**Decision:** Use Model Context Protocol (MCP) вҖ” JSON-RPC 2.0 over stdio. Go binaries run as MCP servers spawned by the Hermes gateway.
23|
24|**Status:** Accepted (2026-05-26)
25|
26|**Consequences:**
27|- вң… Standardized: No custom protocol design needed
28|- вң… Gateway-managed: Lifecycle (spawn, health check, restart) handled by Hermes
29|- вң… Discoverable: Tools are declared and introspectable
30|- вҡ пёҸ Stdio-only: No HTTP/SSE transport вҖ” limits remote access (by design)
31|- вҡ пёҸ Gateway restart needed for MCP binary updates (reload doesn't respawn children)
32|
33|## ADR-003: File-Based Persistence over SQL
34|
35|**Context:** We needed to persist agent state, history, and configuration. SQLite was considered but rejected for simplicity.
36|
37|**Decision:** JSON/YAML files with atomic writes (write .tmp вҶ’ rename) in `~/.go-bt-evolve/`. No database dependency.
38|
39|**Status:** Accepted (2026-05-27)
40|
41|**Consequences:**
42|- вң… Git-friendly: Agent definitions are versionable YAML
43|- вң… Zero dependencies: No database driver or migration tooling
44|- вң… Debuggable: Any text editor can inspect state
45|- вҡ пёҸ No query capability: List/filter operations are O(n) scans
46|- вҡ пёҸ Concurrent writes risk: Mitigated by per-agent file granularity and mutexes
47|
48|## ADR-004: YAML-Defined Agent Platform
49|
50|**Context:** Agents need metadata (name, tree, schedule, I/O contracts, quality gates) separate from the tree definition itself. We needed a registry, scheduler, and catalog.
51|
52|**Decision:** Agents are YAML files in `~/.go-bt-evolve/agents/`. The Registry loads them on startup. The Scheduler runs them on cron schedules. The Catalog provides browsing/installation from templates.
53|
54|**Status:** Accepted (2026-05-27)
55|
56|**Consequences:**
57|- вң… Declarative: Agent config is human-readable YAML
58|- вң… Template marketplace: 24 template agents in `agents/templates/`
59|- вң… Scheduler persistence: FileJobStore survives restarts
60|- вҡ пёҸ Registry is in-memory: All agents loaded at startup (O(n) memory)
61|- вҡ пёҸ No hot-reload: Agent YAML changes require restart or explicit reload
62|
63|## ADR-005: Stockfish-Adapted Evolution Engine
64|
65|**Context:** Behavior trees can degrade with random mutations. We needed an evolution engine that systematically improves trees across multiple fitness dimensions.
66|
67|**Decision:** Adapt Stockfish chess engine techniques вҖ” transposition table for caching evaluated mutations, move ordering by predicted fitness delta, alpha-beta pruning for search. Combine with Pareto front for multi-objective optimization and MAP-Elites for quality diversity.
68|
69|**Status:** Accepted (2026-05-27)
70|
71|**Consequences:**
72|- вң… Six algorithms covering different optimization strategies
73|- вң… Git-versioned: Every accepted mutation is a commit
74|- вң… Reversible: Rollback on regression
75|- вҡ пёҸ 97.3% mutation regression rate: Quality gates needed (see R1 in Section 11)
76|- вҡ пёҸ Per-tree fitness still evolving (reflection.FilterByTreeName)
77|
78|## ADR-006: ChainAction вҖ” LLM Integration via BT Nodes
79|
80|**Context:** LLM calls were initially ad-hoc. We needed them as first-class behavior tree nodes so they benefit from PreGate gating, retry, and StrategyRouter selection.
81|
82|**Decision:** ChainAction nodes wrap LLM calls in the behavior tree. 10 chain types cover single calls, agent loops, RAG, tool use, and multi-step workflows. Configuration is read from node Name and Metadata.
83|
84|**Status:** Accepted (2026-05-28)
85|
86|**Consequences:**
87|- вң… LLM calls are BT-composable: gated, retried, selected
88|- вң… Template variables enable context injection ({{.Task}}, {{.ChainState.*}})
89|- вң… 10 chain types cover diverse LLM workflows
90|- вҡ пёҸ ChainAction panic recovery needed SafeGo wrapper
91|- вҡ пёҸ max_tokens audit detected nodes with max_tokens=1 (aspirational fix)
92|
93|## ADR-007: Reliability Architecture вҖ” Circuit Breakers, Retry, DLQ
94|
95|**Context:** LLM calls fail transiently (Ollama OOM, API rate limits). Goroutines can panic (nil dereference in chain processing). Failed tasks must not be silently lost.
96|
97|**Decision:** Three-layer reliability: SafeGo (panic recovery in all goroutines), CircuitBreaker (3-state: closed/open/half-open with per-agent isolation), RetryWithBackoff (full jitter, 3 classes: standard, LLM, unknown), DeadLetterQueue (persistent JSON for exhausted retries).
98|
99|**Status:** Accepted (2026-05-29)
100|
101|**Consequences:**
102|- вң… Graceful degradation: Single failure doesn't cascade
103|- вң… Failed work preserved: DLQ enables manual inspection and replay
104|- вң… Per-agent circuit breakers: One misbehaving agent doesn't block others
105|- вҡ пёҸ Retry delays add latency (1sвҶ’2sвҶ’4sвҶ’8s backoff)
106|- вҡ пёҸ DLQ grows unbounded without cleanup
